# 法律事務所 AI Agent 實戰課

## 講座定位

2 小時講座，對象以律師為主，場域是福音團契性質的聚會。目標不是介紹 AI 概念，而是讓聽眾看懂三件事：

1. 哪些法律事務所工作可以交給 AI Agent。
2. 哪些步驟一定要保留人工確認。
3. 如何用 30 天做出第一個可用、自動化、可稽核的 AI 助理。

## 核心主軸

法律事務所不是缺 AI 工具，而是缺一套能把「查資料、聯絡、排程、彙整、寄送、追蹤」串起來的流程。AI Agent 的價值不是取代律師或秘書，而是把重複、容易漏、容易出錯的工作先整理好，再交給人確認。

在福音團契語境中，可以把 AI Agent 定位成「忠心管家」的工具：不是為了炫技，而是幫助專業人士更好管理時間、降低行政錯誤，把更多心力留給判斷、陪伴、牧養與真正需要人的工作。

## 主要敘事設定

講座中模擬一間法律事務所：「恩典法律事務所」。它不是大型科技公司，沒有龐大 IT 部門，但有很真實的三種壓力：

1. 律師要掌握最新 AI 與法律議題，但沒有時間每天追資料。
2. 客戶習慣用 LINE 聯絡，秘書需要反覆確認時間與案件資訊。
3. 律師的 Time Record 需要先算出服務時數，再整理成帳單，還要寄給客戶、會計並追蹤付款。

整場講座用這間模擬事務所來示範：如何從「一句需求」變成「一條可運作、可審核、可改進的工作流」。

## 法律倫理與保密邊界

這段需明確說清楚，讓律師聽眾知道 demo 的設計有把專業責任放在核心：

1. 不把真實客戶個資、案件事實、訴訟策略輸入未經核准的外部工具。
2. AI 可以摘要、分類、草擬、提醒，但不能成為最終法律判斷者。
3. 對外文件、客戶通知、帳單、付款狀態都需人工確認後才能送出。
4. 身分分流是必要設計：客戶只能看自己的案件，訪客只看一般資訊，秘書與律師有不同權限。
5. 所有重要流程應保留紀錄：來源、摘要時間、人工確認者、送出時間。

## 過渡案例：每日靈糧

`每日靈糧` 是適合團契場域的主要過渡案例，因為它不是法律系統，但它清楚展示了工作流的本質：固定內容、每日節奏、行動裝置、LINE 分享、雲端同步、提醒與維護。

可講重點：

1. 每天自動依日期顯示內容，不靠人每天手動貼文。
2. 內容可被整理成適合手機閱讀的格式。
3. 可一鍵分享到 LINE，降低傳遞成本。
4. 可做 PWA，像 App 一樣使用。
5. 可用 Cloudflare Worker 做定時 LINE 推播。
6. 對法律事務所的啟發：每日讀經節奏可以轉成每日法律雷達、每週客戶更新、每月帳單提醒。

## 三個法律主案例

### 1. AI 法律資料推播助理

目標：讓律師不用每天自己找資料，但仍能掌握最新 AI 法律議題。

流程：

1. 定時搜尋指定來源。
2. 擷取最新文章、判決、法規、主管機關消息。
3. 去重與分類。
4. 產生 3 層摘要：一句話、律師版、客戶版。
5. 標記來源、日期、可信度與待確認事項。
6. 推播到 email、LINE 或內部看板。

人工確認點：

- 是否真的與事務所業務相關。
- 是否可轉發給客戶。
- 是否需要律師補充法律判斷。

### 2. LINE 排程與秘書協作 Agent

目標：把客戶在 LINE 反覆確認時間的流程整理成半自動工作流。

橋接案例：

- `藍鴨4號` 已經示範「已註冊學生依自己的上課紀錄回答；未註冊訪客只回答一般課程資訊」。
- 法律版使用同樣模式：「已識別客戶看自己的案件與預約；未識別訪客只看一般服務資訊；秘書看待確認卡；律師看需判斷事項」。

流程：

1. 客戶用 LINE 提出需求。
2. Agent 辨識案件、客戶、需求類型與期望時間。
3. 查詢可約時段。
4. 產生 2-3 個候選時間。
5. 發給秘書確認。
6. 秘書核准後，Agent 回覆客戶並建立行事曆事件。

人工確認點：

- 是否為可接受的客戶/案件。
- 是否需要律師親自處理。
- 是否可正式發出邀約。

### 3. 服務時數彙整到帳單草稿

目標：降低秘書整理服務時數、帳單與付款追蹤的錯誤率。

流程：

1. 定時抓取各律師 Time Record。
2. 先依律師彙整服務時數：總時數、可請款時數、不請款/內部時數。
3. 再依客戶與案件彙整服務時數，找出描述太短、單筆過長、案件編號缺漏、不請款項目。
4. 服務時數經律師或秘書確認後，才套用費率與可請款規則。
5. 產生帳單草稿。
6. 主持人或指定人員確認。
7. 確認後 email 給客戶與會計。
8. 後續追蹤付款狀態並回報。

人工確認點：

- 服務時數是否完整。
- 工時描述是否足以讓客戶理解服務內容。
- 是否有不能請款或需折讓項目。
- email 是否可以寄出。
- 付款狀態是否由會計確認。

## 可引用的 repo 故事

- `每日靈糧`：主案例之一，示範固定內容、每日排程、LINE 分享、Cloudflare Worker 定時推播、PWA 與同步，對應「把一個固定節奏做成系統」。
- `booking-system/kefu-bot`：藍鴨4號，示範學生/訪客身分分流、個人化紀錄回答與一般課程資訊，對應「法律事務所的客戶資料隔離」。
- `launchdock`：示範如何把 AI 教學內容做成可互動的學習平台，對應「內部訓練與回饋」。
- `launchdock-lab`：示範 demo 索引、QR code、GitHub Actions 自動建置，對應「把解決方案整理成可展示資產」。
- `dev-harness`：示範多 repo、多 agent 的狀態交接，對應「事務所不能只靠記憶，要有可稽核的交接檔」。

## 結尾轉換

結尾可以輕帶後續服務，但保持低壓：

- 想自己做：可參加工作坊，把今天的模擬事務所流程改成自己的版本。
- 想導入更快：可做顧問盤點，先選一條工作流試作。
- 已有內部系統：可做內部導入案，從資料推播、LINE 排程或帳單草稿三選一開始。

## 展示風格方向

- 現代、清楚、低干擾。
- 用深色背景搭配高對比文字。
- 每一段只呈現一個核心畫面。
- 多用流程圖、時間軸、對話模擬、審核節點。
- 避免大量文字投影片。
